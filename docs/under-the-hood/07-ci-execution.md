# CI Execution

How `uvr build` and `uvr finalize` dispatch pre-computed commands in GitHub
Actions, and what each CI job does.

See [How it works](../user-guide/09-architecture.md) for the high-level flow.

## Source files

| Module | Key symbols |
|--------|-------------|
| `execute.py` | `ReleaseExecutor` (`build`, `publish`, `finalize`, `run`, `_run_stage`, `_run_commands`) |
| `models.py` | `BuildStage`, `PlanCommand`, `_BUILD_STEPS`, `_PUBLISH_STEPS`, `_FINALIZE_STEPS`, `_CHECKOUT_STEP`, `_SETUP_UV_STEP`, `_SETUP_PYTHON_STEP`, `_EXPORT_PLAN_STEP` |
| `plan.py` | `ReleasePlanner` (`_generate_build_commands`, `_generate_publish_commands`, `_generate_finalize_commands`) |
| `cli/__init__.py` | `_cmd_build`, `_cmd_finalize` (CLI wiring) |

## The `uvr` CLI entry point

The `uvr` console script is registered in `cli/__init__.py`. Two subcommands
are used by CI:

| Subcommand | Handler | Arguments |
|------------|---------|-----------|
| `build` | `_cmd_build` | `--plan`, `--runner` |
| `finalize` | `_cmd_finalize` | `--plan` |

Both handlers deserialize the plan with `ReleasePlan.model_validate_json(plan)`
and delegate to `ReleaseExecutor`.

```python
def _cmd_build(a):
    plan = ReleasePlan.model_validate_json(a.plan)
    ReleaseExecutor(plan).build(runner=a.runner)

def _cmd_finalize(a):
    plan = ReleasePlan.model_validate_json(a.plan)
    ReleaseExecutor(plan).finalize()
```

## The `fromJSON(inputs.plan)` pattern

The workflow file passes the plan JSON as a `workflow_dispatch` input:

```yaml
on:
  workflow_dispatch:
    inputs:
      plan:
        description: Release plan JSON (generated by uvr release).
        type: string
        required: true
```

Within the workflow YAML, plan fields are accessed via:

```yaml
${{ fromJSON(inputs.plan).field_name }}
```

This is used for:
- **Job `if` conditions**: `!contains(fromJSON(inputs.plan).skip, 'build')`
- **Matrix expansion**: `strategy.matrix.runner: ${{ fromJSON(inputs.plan).runners }}`
- **Publish matrix**: `strategy.matrix.include: ${{ fromJSON(inputs.plan).publish_matrix }}`
- **Python version**: `uv python install ${{ fromJSON(inputs.plan).python_version }}`
- **uvr install spec**: `uv tool install ${{ fromJSON(inputs.plan).uvr_install }}`

The `_P` constant in `models.py` (`"fromJSON(inputs.plan)"`) is a shorthand used
to construct these expressions in Python without repeating the full string.

## Workflow step blocks

### Shared preamble steps

Every job that runs uvr commands uses these steps:

1. **`_CHECKOUT_STEP`**: `actions/checkout@v4` with `fetch-depth: 0` (full
   history needed for git tags and diffs during finalize).

2. **`_SETUP_UV_STEP`**: `astral-sh/setup-uv@v5` to install the `uv` tool.

3. **`_SETUP_PYTHON_STEP`**: Runs `uv python install` with the plan's Python
   version and `uv tool install` with the plan's uvr install spec.

### Build job (`_BUILD_STEPS`)

```
1. Checkout (full history)
2. Setup uv
3. Setup Python + install uvr
4. Build packages:
     uvr build --plan "$UVR_PLAN" --runner '${{ toJSON(matrix.runner) }}'
5. Upload artifact:
     actions/upload-artifact@v4 -> wheels-${{ join(matrix.runner, '-') }}
```

The build step invokes `ReleaseExecutor.build()`, which runs pre-computed
`BuildStage` lists (see [Build matrix](04-build-matrix.md)).

The artifact name is `wheels-${{ join(matrix.runner, '-') }}`, so each runner
uploads its own artifact. The publish job later downloads all of them with
`pattern: wheels-*` and `merge-multiple: true`.

### Publish job (`_PUBLISH_STEPS`)

```
1. Download artifacts:
     actions/download-artifact@v4
       pattern: wheels-*
       merge-multiple: true
       run-id: <plan.reuse_run_id or github.run_id>
2. Create GitHub release:
     softprops/action-gh-release@v2
       tag_name: ${{ matrix.tag }}
       name: ${{ matrix.title }}
       body: ${{ matrix.body }}
       files: ${{ matrix.dist_name }}-${{ matrix.version }}*.whl
       make_latest: ${{ matrix.make_latest }}
```

The publish job runs as a matrix over `plan.publish_matrix`. Each entry provides
the tag, title, body, and dist name. The `files` glob matches only the wheels
for that specific package. No `uvr` command is invoked -- the
`softprops/action-gh-release` action handles publishing directly.

The `run-id` expression handles artifact reuse:
```
${{ fromJSON(inputs.plan).reuse_run_id != '' && fromJSON(inputs.plan).reuse_run_id || github.run_id }}
```

### Finalize job (`_FINALIZE_STEPS`)

```
1. Checkout (full history)
2. Setup uv
3. Setup Python + install uvr
4. Finalize:
     uvr finalize --plan "$UVR_PLAN"
```

## `ReleaseExecutor.finalize()` -- the finalize phase

`ReleaseExecutor.finalize()` calls `_run_commands()` on the pre-computed
`plan.finalize_commands` list. These commands are generated by
`ReleasePlanner._generate_finalize_commands()` at plan time. The executor
contains no domain logic -- it just runs `subprocess.run()` on each
`PlanCommand`.

The finalize command list performs these steps in order:

1. **Configure git identity** (when `ci_publish=True`):
   ```
   git config user.name "github-actions[bot]"
   git config user.email "github-actions[bot]@users.noreply.github.com"
   ```

2. **Tag releases** (when `ci_publish=False` only): creates `{name}/v{version}`
   tags. In CI mode, the `softprops/action-gh-release` action already created
   the tags during the publish step.

3. **Apply version bumps**: runs `uv version {next_version} --directory {path}`
   for each changed package, writing the next `.devN` version.

4. **Pin internal deps**: runs `uvr pin-deps --path {pyproject} DEP>=VER` for
   each package with internal dependencies, pinning to just-published versions.

5. **Sync and commit**: runs `uv sync --all-groups --all-extras`, stages
   modified `pyproject.toml` files and `uv.lock`, and commits with message
   `"chore: prepare next release"`.

6. **Tag baselines**: creates `{name}/v{new_version}-base` tags on the bump
   commit.

7. **Push** (when `ci_publish=True`):
   ```
   git push
   git push --tags
   ```

### Data flow

```
ReleaseExecutor.finalize()
  -> _run_commands(plan.finalize_commands)
  -> for each PlanCommand:
       subprocess.run(cmd.args)
       (abort on failure if cmd.check is True)
```

The commands themselves (generated at plan time):

```
_generate_finalize_commands(changed, bumps, published_versions)
  -> (if ci_publish) git config identity
  -> (if !ci_publish) git tag {name}/v{version} for each changed package
  -> uv version {next_version} for each bump
  -> uvr pin-deps for each package with internal deps
  -> uv sync --all-groups --all-extras
  -> git add {pyproject files} uv.lock
  -> git commit -m "chore: prepare next release"
  -> git tag {name}/v{new_version}-base for each bump
  -> (if ci_publish) git push + git push --tags
```

## `ReleaseExecutor.build()` -- the build phase

See [Build matrix](04-build-matrix.md) for the full algorithm. Key points:

- Selects `BuildStage` list for the current `--runner` from
  `plan.build_commands`.
- Stages execute sequentially; packages within a stage execute in parallel via
  `ThreadPoolExecutor`.
- Stage 0 is always setup (mkdir dist, fetch unchanged wheels).
- Middle stages correspond to topological layers (layer 0, layer 1, ...).
- Final stage (if present) is cleanup -- removes wheels for transitive deps not
  assigned to this runner.

## `ReleaseExecutor.run()` -- convenience wrapper

`ReleaseExecutor.run()` chains all three phases sequentially for local use:

```python
def run(self) -> None:
    self.build()
    self.publish()
    self.finalize()
```

This is the code path used by `uvr release --where local` and
`uvr release --plan`. In CI, these phases are separate jobs with `needs`
dependencies.

## Environment variable passing

The `_EXPORT_PLAN_STEP` (defined but used optionally) exports plan data to
`$GITHUB_ENV`:

```bash
echo "UVR_CHANGED=$(echo "$UVR_PLAN" | jq -r '.changed | keys | join(" ")')" >> "$GITHUB_ENV"
echo "UVR_PLAN=$UVR_PLAN" >> "$GITHUB_ENV"
```

This makes `$UVR_CHANGED` (space-separated list of changed package names) and
`$UVR_PLAN` available to subsequent steps without re-parsing the input.

## Hook jobs

The workflow supports optional hook jobs that run between the core pipeline
stages. These appear in `release.yml` between the core jobs:

| Hook job | Runs between | Default |
|----------|-------------|---------|
| `pre-build` | (start) and `build` | lint + typecheck + test |
| `post-build` | `build` and `publish` | no-op |
| `pre-release` | `post-build` and `publish` | no-op |
| `post-release` | `finalize` and (end) | PyPI publish for `uv-release-monorepo` |

Hook jobs are user-editable. They can be skipped via the plan's `skip` list
(e.g., `--skip pre-build`).
