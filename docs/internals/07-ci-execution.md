# CI Execution

How `uvr-steps` dispatches commands in GitHub Actions and what each CI step does.

See [How it works](../guide/09-architecture.md) for the high-level flow.

## Source files

| Module | Key functions |
|--------|---------------|
| `workflow_steps.py` | `main`, `execute_build_all`, `execute_finalize`, `execute_build`, `execute_prepare_release`, `execute_fetch_unchanged`, `execute_publish_releases`, `execute_release` |
| `models.py` | `_BUILD_STEPS`, `_PUBLISH_STEPS`, `_FINALIZE_STEPS`, `_CHECKOUT_STEP`, `_SETUP_UV_STEP`, `_SETUP_PYTHON_STEP`, `_EXPORT_PLAN_STEP` |
| `pipeline.py` | `build_packages`, `fetch_unchanged_wheels`, `publish_release`, `tag_changed_packages`, `apply_bumps`, `commit_bumps`, `tag_dev_baselines` |

## The `uvr-steps` entry point

`workflow_steps.py:main` is an argparse-based CLI registered as the `uvr-steps`
console script. It dispatches to the following subcommands:

| Subcommand | Handler | Arguments |
|------------|---------|-----------|
| `build-all` | `execute_build_all` | `--plan`, `--runner` |
| `execute-build` | `execute_build` | `--plan`, `--package` |
| `prepare-release` | `execute_prepare_release` | `--plan`, `--package` |
| `fetch-unchanged` | `execute_fetch_unchanged` | `--plan` |
| `publish-releases` | `execute_publish_releases` | `--plan` |
| `finalize` | `execute_finalize` | `--plan` |
| `execute-release` | `execute_release` | `--plan` |

Every handler receives the plan as a raw JSON string and deserializes it with
`ReleasePlan.model_validate_json(plan_json)`.

## The `fromJSON(inputs.plan)` pattern

The workflow file passes the plan JSON as a `workflow_dispatch` input:

```yaml
on:
  workflow_dispatch:
    inputs:
      plan:
        description: Release plan JSON (generated by uvr release).
        type: string
        required: true
```

Within the workflow YAML, plan fields are accessed via:

```yaml
${{ fromJSON(inputs.plan).field_name }}
```

This is used for:
- **Job `if` conditions**: `!contains(fromJSON(inputs.plan).skip, 'build')`
- **Matrix expansion**: `strategy.matrix.runner: ${{ fromJSON(inputs.plan).runners }}`
- **Publish matrix**: `strategy.matrix.include: ${{ fromJSON(inputs.plan).publish_matrix }}`
- **Python version**: `uv python install ${{ fromJSON(inputs.plan).python_version }}`
- **uvr install spec**: `uv tool install ${{ fromJSON(inputs.plan).uvr_install }}`

The `_P` constant in `models.py` (`"fromJSON(inputs.plan)"`) is a shorthand used
to construct these expressions in Python without repeating the full string.

## Workflow step blocks

### Shared preamble steps

Every job that runs uvr commands uses these steps:

1. **`_CHECKOUT_STEP`**: `actions/checkout@v4` with `fetch-depth: 0` (full
   history needed for git tags and diffs during finalize).

2. **`_SETUP_UV_STEP`**: `astral-sh/setup-uv@v5` to install the `uv` tool.

3. **`_SETUP_PYTHON_STEP`**: Runs `uv python install` with the plan's Python
   version and `uv tool install` with the plan's uvr install spec.

### Build job (`_BUILD_STEPS`)

```
1. Checkout (full history)
2. Setup uv
3. Setup Python + install uvr
4. Build packages:
     uvr-steps build-all --plan "$UVR_PLAN" --runner '${{ matrix.runner }}'
5. Upload artifact:
     actions/upload-artifact@v4 -> wheels-{runner}
```

The build step invokes `execute_build_all`, which handles the full per-runner
build logic (see [Build matrix](04-build-matrix.md)).

The artifact name is `wheels-${{ matrix.runner }}`, so each runner uploads its
own artifact. The publish job later downloads all of them with
`pattern: wheels-*` and `merge-multiple: true`.

### Publish job (`_PUBLISH_STEPS`)

```
1. Download artifacts:
     actions/download-artifact@v4
       pattern: wheels-*
       merge-multiple: true
       run-id: <plan.reuse_run_id or github.run_id>
2. Create GitHub release:
     softprops/action-gh-release@v2
       tag_name: ${{ matrix.tag }}
       name: ${{ matrix.title }}
       body: ${{ matrix.body }}
       files: ${{ matrix.dist_name }}-${{ matrix.version }}*.whl
       make_latest: ${{ matrix.make_latest }}
```

The publish job runs as a matrix over `plan.publish_matrix`. Each entry provides
the tag, title, body, and dist name. The `files` glob matches only the wheels
for that specific package.

The `run-id` expression handles artifact reuse:
```
${{ fromJSON(inputs.plan).reuse_run_id != '' && fromJSON(inputs.plan).reuse_run_id || github.run_id }}
```

### Finalize job (`_FINALIZE_STEPS`)

```
1. Checkout (full history)
2. Setup uv
3. Setup Python + install uvr
4. Finalize:
     uvr-steps finalize --plan "$UVR_PLAN"
```

## `execute_finalize` -- the finalize step

`workflow_steps.py:execute_finalize` performs all post-publish work:

1. **Configure git identity** (when `ci_publish=True`):
   ```
   git config user.name "github-actions[bot]"
   git config user.email "github-actions[bot]@users.noreply.github.com"
   ```

2. **Skip release tagging** (when `ci_publish=True`): The `softprops/action-gh-release`
   action already created the tags during the publish step.

3. **Apply version bumps**: `pipeline.py:apply_bumps` reads `plan.bumps` and
   writes `make_dev(bump.new_version)` to each package's `pyproject.toml`.

4. **Commit**: `pipeline.py:commit_bumps` stages modified `pyproject.toml` files,
   runs `uv sync` to update `uv.lock`, and commits with message
   `"chore: prepare next release"`.

5. **Tag dev baselines**: `pipeline.py:tag_dev_baselines` creates
   `{name}/v{new_version}-dev` tags on the bump commit.

6. **Push** (when `ci_publish=True`):
   ```
   git push
   git push --tags
   ```

### Data flow

```
execute_finalize(plan_json)
  -> ReleasePlan.model_validate_json(plan_json)
  -> (if ci_publish) git config identity
  -> (if !ci_publish) tag_changed_packages(plan.changed)
  -> apply_bumps(plan) -> bumped dict
  -> commit_bumps(plan.changed, bumped)
  -> tag_dev_baselines(bumped)
  -> (if ci_publish) git push + git push --tags
```

## `execute_build_all` -- the build step

See [Build matrix](04-build-matrix.md) for the full algorithm. Key points:

- Builds only packages assigned to the current `--runner`.
- Fetches wheels for unchanged transitive deps.
- Builds changed packages in topological order.
- Cleans up non-assigned wheels before artifact upload.

## `execute_release` -- convenience wrapper

`execute_release` chains three steps sequentially for local use:

```python
def execute_release(plan_json: str) -> None:
    execute_fetch_unchanged(plan_json)
    execute_publish_releases(plan_json)
    execute_finalize(plan_json)
```

This is the code path used by `uvr run --plan`. In CI, these steps are separate
jobs with `needs` dependencies.

## `execute_prepare_release`

Strips the `.dev` suffix from a single package's `pyproject.toml`. This is a
legacy step for per-package build jobs (before `build-all` existed). It writes
the clean release version so `uv build` produces a wheel with the correct
version number.

## Environment variable passing

The `_EXPORT_PLAN_STEP` (defined but used optionally) exports plan data to
`$GITHUB_ENV`:

```bash
echo "UVR_CHANGED=$(echo "$UVR_PLAN" | jq -r '.changed | keys | join(" ")')" >> "$GITHUB_ENV"
echo "UVR_PLAN=$UVR_PLAN" >> "$GITHUB_ENV"
```

This makes `$UVR_CHANGED` (space-separated list of changed package names) and
`$UVR_PLAN` available to subsequent steps without re-parsing the input.
